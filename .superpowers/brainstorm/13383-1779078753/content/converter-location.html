Where should STEP analysis/conversion run?

This affects privacy, dependency risk, and how deeply we can inspect assembly labels.

A

In-browser worker using OpenCascade.js Recommended

Add converter logic near current STEP import path, inside cascade-core worker APIs.

Pros

  • Fits CascadeStudio architecture: CAD parsing already happens in Web Worker + WASM.
  • Private/offline: user STEP files do not leave browser.
  • Reuses existing build/deploy model; no server dependency for GitHub Pages/Vercel static app.
  • Generated code can reference same externalShapes state immediately.

Cons

  • Limited by OpenCascade.js bindings available in browser.
  • Heavy assembly traversal may block worker longer; needs progress/errors.
  • Harder to use native OCCT/XDE tools if bindings are missing.
B

Local CLI converter

Provide a Node/native script that reads STEP and outputs a CascadeStudio project/code package.

Pros

  • Can use stronger native OCCT/XDE tooling if browser bindings fall short.
  • Better for batch conversion and very large assemblies.
  • Keeps heavy processing outside interactive browser session.

Cons

  • Does not fit casual web app workflow; users must install/run tools.
  • More packaging/support burden across macOS/Windows/Linux.
  • Generated project still needs import path compatibility with browser app.
C

Hosted conversion service

Upload STEP to backend service, return generated CascadeStudio JS/project.

Pros

  • Backend can use native libraries and more CPU/memory.
  • Small browser code; easy to update conversion logic centrally.
  • Can add queueing, batch jobs, and cached conversions later.

Cons

  • Privacy/IP concern for user CAD files.
  • Breaks static/offline app promise and adds hosting cost.
  • Needs auth, upload limits, storage lifecycle, security hardening.
