Where should the STEP data live in generated JS?

This controls shareability, project size, and whether generated code works without extra files.

A

Inline STEP text in the JS

Generated code contains the full STEP file as a string or compressed/base64 blob, then imports it at runtime.

Pros

  • Single copy-paste artifact; user can share one CascadeStudio URL/project.
  • No dependency on uploaded file names or browser file persistence.
  • Good for small vendor parts and examples.

Cons

  • Large assemblies can bloat Monaco, URL serialization, and browser memory.
  • Generated code becomes noisy; hard for users to read actual assembly logic.
  • May hit URL/localStorage limits if auto-shared.
B

Attach STEP as project external file Recommended

Keep STEP content in CascadeStudio’s existing external file/project storage, and generate clean JS that references it by name.

Pros

  • Fits current app model: imported files already populate externalShapes[fileName].
  • Keeps generated JS readable: assembly code, names, transforms, comments stay visible.
  • Better for larger assemblies than inline code or URL payloads.
  • Lowest implementation risk: extends current load/save project behavior.

Cons

  • Generated JS alone is not enough; project must retain attached file data.
  • Needs clear missing-file errors if user pastes code into fresh session.
  • Sharing raw code outside project package loses geometry.
C

Reference external URL or path

Generated code fetches STEP data from a user-provided URL, server asset, or local path-like reference.

Pros

  • Smallest generated code and project metadata.
  • Works well for hosted libraries of reusable hardware/components.
  • Can update geometry without regenerating JS.

Cons

  • Browser CORS/offline constraints make this brittle for casual users.
  • Local paths cannot be read from browser JS without user file selection.
  • Remote asset changes can silently alter rendered model.
