What should generated CascadeStudio JS look like?

This controls whether non-coding users can safely edit parts after conversion.

A

Declarative part manifest + small helper functions Recommended

Generated code starts with editable data objects, then generic helper code loads, colors, transforms, hides, and groups parts.

Pros

  • Best fit for non-coders: edit names, colors, visibility, transforms in one clear table.
  • Reliable: helper can enforce missing-file checks and stable fallbacks around externalShapes.
  • Maintainable: converter changes manifest schema without rewriting lots of imperative code.
  • YAGNI-friendly: no new UI required for first version.

Cons

  • Generated JS includes boilerplate helper code unless helpers become built-in library functions.
  • Users must understand simple object fields like visible, color, translate.
  • Complex conditional assembly logic still needs real JS edits.
B

Straight imperative JS

Emit simple sequential lines: get shape, translate, rotate, color, push to scene for each part.

Pros

  • Looks like normal CascadeStudio examples and is easy to debug line-by-line.
  • No schema docs needed; each operation is explicit JS.
  • Good for power users who already know Translate(), Rotate(), sceneShapes.

Cons

  • Large assemblies become repetitive and intimidating for non-coders.
  • Harder to bulk rename, recolor, hide, or parameterize parts safely.
  • Missing-file and fallback logic gets duplicated or omitted.
C

Generated GUI controls first

Emit code that creates Tweakpane controls for every part: visibility toggles, colors, transforms, group switches.

Pros

  • Most accessible for users who avoid code entirely.
  • Uses existing CascadeStudio GUI/Tweakpane pattern.
  • Good for exploring assemblies visually after import.

Cons

  • Can overwhelm GUI panel for many parts.
  • Does not solve clean source-of-truth unless backed by a manifest anyway.
  • More generated code and performance risk before core conversion is stable.
