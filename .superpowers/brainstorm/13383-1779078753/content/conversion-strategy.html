What should “convert STEP to CascadeStudio JS” mean first?

STEP can be rendered directly, but fully rebuilding editable parametric code from B-Rep geometry is not reliable. Pick target output.

A

Import wrapper JS + extracted assembly map Recommended

Generate CascadeStudio JS that embeds/loads STEP content, names parts, applies transforms/colors, and exposes user-editable hooks.

Pros

  • Most reliable: reuses OCCT STEP reader already in FileUtils.js.
  • Preserves exact CAD geometry instead of guessing primitives from B-Rep faces.
  • Fits CascadeStudio today: externalShapes, sceneShapes.push(), existing STEP import.
  • Lets users edit assembly metadata without needing CAD kernel internals.

Cons

  • Output is not fully parametric reconstruction like Box()/FilletEdges().
  • Big STEP files make generated JS/project storage large unless external asset kept.
  • Feature history stays opaque: cannot edit original sketches/fillets.
B

Best-effort primitive reconstruction

Analyze STEP geometry and emit CascadeStudio operations like cylinders, boxes, revolves, booleans, and transforms where recognized.

Pros

  • Could produce readable, editable CascadeStudio code for simple mechanical parts.
  • Smaller output for primitive-heavy models.
  • Useful as learning scaffold for users who want to modify design intent.

Cons

  • Fragile: STEP stores final B-Rep, not original modeling history.
  • Hard to infer boolean order, sketches, fillet selectors, and design parameters.
  • High maintenance risk in browser/WASM worker; many false reconstructions.
C

Hybrid: exact STEP baseline + optional recognized helpers

Always render exact imported shapes, then annotate likely cylinders/planes/axes and generate optional helper code for user replacement.

Pros

  • Reliable viewport path remains exact even when recognition fails.
  • Gives users part names, axes, dimensions, and editable hints over time.
  • Good incremental roadmap: ship import wrapper first, add recognizers later.

Cons

  • More UX complexity: generated code has exact imports plus advisory helper blocks.
  • Requires clear warnings so users know which code is authoritative.
  • Recognizer output still may be incomplete for organic or vendor CAD.
